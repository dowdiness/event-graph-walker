# Baseline Benchmarks — 2026-02-28

Recorded from `moon bench --release` on the `text` package.
Platform: Linux 6.6.87.2-microsoft-standard-WSL2

## How to reproduce

```bash
cd event-graph-walker
moon bench --package text --release
```

---

## Results

### Insert

| Benchmark | Mean | Std Dev |
|---|---|---|
| insert append (100 chars) | 9.31 ms | ± 403 µs |
| insert append (1000 chars) | 8.76 s | ± 156 ms |
| insert prepend (100 chars) | 9.11 ms | ± 177 µs |

**Note:** Insert is O(n²) in total setup cost. Going from 100 → 1000 chars (10×) costs ~940× more time.
The hot path is the full `position_cache` rebuild on every mutation: the cache is a lazily-evaluated
`Array[(Lv, Item[String])]?` invalidated on each insert/delete, so rebuilding it on the *next* read
costs O(n) and happens once per loop iteration.

### Delete

| Benchmark | Mean | Std Dev |
|---|---|---|
| delete (100 deletes from 100-char doc) | 34.17 ms | ± 1.50 ms |

**Note:** ~3.4× slower than 100 appends (9.31 ms) because each delete must also walk the tree to
translate a visible-position offset, and the cache is invalidated after every delete.

### Read — `text()` and `len()`

| Benchmark | Mean | Std Dev |
|---|---|---|
| text() — 100-char doc | 127.95 µs | ± 8.53 µs |
| text() — 1000-char doc | 15.33 ms | ± 831 µs |
| len() — 1000-char doc | 13.95 ms | ± 230 µs |

**Note:** `len()` is documented as O(1) via a cached `visible_count`, but it benchmarks at 13.95 ms
for a 1000-char doc — nearly the same as `text()`. Investigate whether `TextDoc::len()` is actually
hitting the tree walk rather than the cached counter.

### Sync — `export_all`, `export_since`, `apply`

| Benchmark | Mean | Std Dev |
|---|---|---|
| export_all — 100 ops | 0.21 µs | ± 0.01 µs |
| export_all — 1000 ops | 1.63 µs | ± 0.14 µs |
| export_since — 50-op delta, 1000-op base | 505.93 µs | ± 14.19 µs |
| sync apply — 50 remote ops | 89.00 µs | ± 3.77 µs |
| sync apply — 500 remote ops | 1.35 ms | ± 49.87 µs |
| bidirectional sync — 2 peers × 50 ops | 174.99 µs | ± 5.15 µs |

**Notes:**
- `export_all` is extremely fast (~1.63 ns/op at 1000 ops) — just a slice copy of the oplog.
- `sync apply` costs ~1.78 µs/op (89 µs ÷ 50).
- `export_since` at 506 µs for a 50-op delta over a 1000-op document reflects the graph diff
  (transitive closure traversal) rather than the export itself.

### Checkout

| Benchmark | Mean | Std Dev |
|---|---|---|
| checkout — midpoint of 100-op doc | 41.97 µs | ± 1.01 µs |
| checkout — midpoint of 1000-op doc | 761.20 µs | ± 29.72 µs |

**Note:** Checkout cost scales roughly linearly with document size (~420 ns/op at 100 ops,
~760 ns/op at 1000 ops). The branch replay dominates.

### Undo / Redo

| Benchmark | Mean | Std Dev |
|---|---|---|
| record_insert — 100 ops, 1 group | 2.19 µs | ± 0.27 µs |
| record_insert — 100 ops, 100 groups | 3.17 µs | ± 0.19 µs |
| undo() — 10-op group | 54.33 µs | ± 1.97 µs |
| undo() — 50-op group | 3.24 ms | ± 102 µs |
| undo + redo roundtrip — 10-op group | 60.44 µs | ± 1.77 µs |
| 10 undo+redo cycles — 10-op group | 426.69 µs | ± 16.33 µs |

**Notes:**
- `record_insert` is cheap (~22–32 ns/op) regardless of group count.
- `undo()` scales quadratically with group size: 54 µs (10 ops) vs 3.24 ms (50 ops) is ~60×
  for a 5× increase, suggesting an O(n²) path in the undo replay.
- Each undo+redo cycle for the 10-op group costs ~43 µs.

## Summary of performance concerns

| Issue | Affected operations | Severity |
|---|---|---|
| O(n²) total insert cost | `insert` in a loop | High — 8.76 s for 1000 chars |
| `len()` not O(1) | `TextDoc::len()` | Medium — investigate |
| O(n²) undo replay | `undo()` on large groups | Medium — 3.24 ms for 50 ops |
| `export_since` graph diff | `SyncSession::export_since` | Low — 506 µs acceptable for most use |

See [`docs/OPTIMIZATION_ROADMAP.md`](../OPTIMIZATION_ROADMAP.md) for planned improvements.
